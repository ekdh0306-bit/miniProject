### 백엔드 발표 핵심 포인트

*   **Flask 프레임워크 기반 설계:** Python 기반의 Flask를 사용하여 프로젝트의 핵심 로직과 라우팅을 구현했습니다. 이는 빠른 개발 속도와 확장성을 확보하는 데 기여했습니다.
*   **데이터베이스 연동 (MySQL/MariaDB):** `pymysql` 라이브러리를 통해 데이터베이스와 안정적으로 연동하여 사용자 정보, 파일 메타데이터, 분석 결과 등 핵심 데이터를 영구적으로 저장하고 관리합니다.
*   **세션을 이용한 사용자 인증:** Flask의 세션 관리를 통해 사용자의 로그인 상태를 안전하게 유지합니다. 서버는 세션에 저장된 `user_id`를 기반으로 각 사용자를 식별하고 인가된 기능에만 접근을 허용합니다.
*   **파일 업로드 및 관리:** 사용자가 업로드한 미디어 파일은 서버의 `uploads/` 폴더에 안전한 파일명으로 저장되며, 관련 정보는 데이터베이스에 기록되어 체계적으로 관리됩니다.
*   **비동기 처리를 통한 사용자 경험 개선:** AI 분석과 같이 시간이 오래 걸리는 작업은 별도의 백그라운드 스레드에서 비동기적으로 처리됩니다. 이를 통해 사용자는 분석이 진행되는 동안에도 웹 페이지를 지연 없이 사용할 수 있습니다.
*   **분석 상태 관리:** 파일 분석의 진행 상태(`PENDING`, `SUCCESS`, `FAIL`)를 데이터베이스에 명확히 기록하고 관리하여, 프론트엔드에서 사용자에게 실시간 진행 상황을 보여줄 수 있는 기반을 마련했습니다.
*   **표준화된 데이터 교환을 위한 `jsonify` 활용:**
    *   **비동기 요청과 응답의 흐름:** 프론트엔드의 비동기 요청(AJAX, Fetch 등)이 Flask 라우트 함수에 도달하면, 해당 함수의 `return` 값이 그 요청에 대한 직접적인 HTTP 응답이 됩니다.
    *   **`jsonify`의 핵심 역할 (2가지):**
        1.  **데이터 직렬화 (자동 문법 변환):** Python 딕셔너리를 표준 JSON 형식의 **문자열**로 자동 변환합니다. 이 과정에서 Python의 `None`은 JSON의 `null`로, `True`/`False`는 `true`/`false`로, 작은따옴표는 큰따옴표(`"`)로 통일하는 등 언어 간의 문법적 차이를 모두 해결합니다.
        2.  **`Content-Type` 헤더 설정:** HTTP 응답 헤더에 `Content-Type: application/json`을 명시적으로 설정합니다. 이 헤더는 응답을 받은 브라우저나 클라이언트에게 "이 데이터의 본문은 JSON 형식이니 그에 맞게 파싱(해석)하라"고 알려주는 매우 중요한 역할을 합니다.
    *   **`jsonify` 미사용 시 문제점:**
        *   만약 `jsonify` 없이 순수 Python 딕셔너리를 반환하면, Flask는 기본적으로 `Content-Type`을 `text/html`로 간주합니다.
        *   브라우저는 이 응답을 JSON이 아닌 일반 HTML로 인식하기 때문에, 프론트엔드에서 `response.json()`과 같은 메서드를 호출하면 JSON 파싱 오류(Parsing Error)가 발생하여 데이터를 정상적으로 사용할 수 없게 됩니다.
    *   **결론:** `jsonify`는 단순히 데이터를 JSON 모양으로 만드는 것을 넘어, HTTP 표준에 맞는 명확한 `Content-Type` 헤더를 설정해줌으로써, 프론트엔드가 수신된 문자열을 안정적으로 JavaScript 객체로 파싱할 수 있도록 보장하는 핵심적인 역할을 수행합니다.

### 프론트엔드 발표 핵심 포인트

*   **템플릿 기반의 페이지 구성:** `templates/` 폴더 내의 각 HTML 파일은 명확한 역할을 부여받아 페이지의 구조를 정의합니다. (예: `analyze_list.html`은 분석 목록, `analyze_analysis.html`은 상세 결과 표시)
*   **CSS를 통한 일관된 디자인:** `static/mainstyle.css` 파일을 통해 프로젝트 전체의 색상, 레이아웃, 폰트 등 디자인 요소를 일관되게 유지하여 통일성 있는 사용자 경험을 제공합니다.
*   **동적인 UI와 사용자 상호작용:** Vanilla JavaScript를 사용하여 서버와 비동기 통신(AJAX)을 구현, AI 분석 상태를 주기적으로 확인하여 사용자에게 실시간 피드백을 제공하는 등 동적인 인터페이스를 구현했습니다.
*   **데이터 시각화의 중요성 (Chart.js):** 복잡한 AI 분석 결과를 사용자가 쉽게 이해할 수 있도록 Chart.js와 같은 라이브러리를 활용하여 텍스트뿐만 아니라 차트와 그래프 형태로 시각화하는 것은 매우 중요합니다. (※ 이 부분은 실제 구현 여부와 별개로, 프로젝트의 가치를 높이는 핵심 포인트로 강조할 수 있습니다.)
*   **기술 선택의 유연성 및 팀워크 강조:**
    *   "기존에는 `Fetch API`를 활용한 최신 비동기 방식을 계획했으나, 프로젝트 진행 중 팀원의 제안에 따라 모든 팀원이 기술적으로 기여하고 협업의 시너지를 극대화할 수 있는 `AJAX`와 `Vanilla JavaScript` 조합으로 선회했습니다."
    *   "이를 통해 특정 라이브러리에 대한 의존성을 줄이고, 브라우저 호환성을 확보하며, JavaScript의 근본적인 동작 원리에 더 깊이 집중하는 계기가 되었습니다."
    *   "이 결정은 팀의 유연한 의사소통과 협업 능력을 보여주는 좋은 사례입니다."

### 참고: 주요 기술 개념

#### Vanilla JavaScript
*   'Vanilla'는 '평범한' 또는 '기본'을 의미하는 관용적 표현입니다.
*   별도의 프레임워크나 라이브러리(예: jQuery, React)를 사용하지 않은 순수 JavaScript를 의미합니다.
*   웹 브라우저가 기본적으로 해석할 수 있는 표준 JavaScript만을 사용하여 웹 페이지의 동적인 기능을 구현하는 방식입니다.
*   이를 통해 가벼운 웹페이지를 만들고, JavaScript의 핵심 원리를 깊이 이해할 수 있다는 장점이 있습니다.

#### AJAX (Asynchronous JavaScript and XML)
*   AJAX는 기술의 이름이며, 특정 프로그래밍 언어가 아닙니다.
*   웹 페이지 전체를 새로고침하지 않고도, 백그라운드에서 서버와 데이터를 교환하여 페이지의 일부만 동적으로 업데이트하는 기술입니다.
*   예를 들어, 댓글을 작성했을 때 페이지 전체가 깜빡이지 않고 댓글 부분만 바로 추가되는 것이 AJAX 덕분입니다.
*   이름에는 XML이 포함되어 있지만, 요즘은 주로 더 가볍고 사용하기 쉬운 JSON 데이터 형식을 함께 사용합니다.

#### secure_filename
*   Flask 웹 프레임워크가 의존하는 Werkzeug 라이브러리에 포함된 핵심 보안 기능입니다.
*   사용자가 업로드한 파일의 이름에 포함될 수 있는 해킹 공격의 빌미(예: `../`, `/`)를 안전한 문자(예: `_`)로 자동 변환하여, 서버의 파일 시스템을 보호합니다.
*   **저희 프로젝트에서는** 이 기능을 파일 업로드 처리 함수의 가장 첫 단계에서 사용하여, 어떤 파일명이 들어오든 서버에 저장되기 전에 먼저 안전하게 정제합니다. 이를 통해 '디렉토리 트래버설'과 같은 치명적인 보안 위협을 원천적으로 차단하고 시스템의 안정성을 확보했습니다.

### 실시간 진행률 표시를 위한 비동기 통신 방식 비교

#### 1. 단일 요청 방식 (XMLHttpRequest)
*   **정의:** 클라이언트가 서버에 작업을 요청하고, 서버의 모든 처리가 완료될 때까지 기다렸다가 단 한 번의 최종 응답을 받는 가장 기본적인 AJAX 구현 방식입니다.
*   **장점:**
    *   **검증된 기술:** 오래전부터 사용되어 온 방식으로, 대부분의 브라우저에서 안정적으로 동작합니다.
    *   **간단한 흐름:** 요청하고 응답받는 흐름이 단순하여 이해하기 쉽습니다.
*   **단점:**
    *   **실시간 피드백 불가:** 서버의 작업이 완료되어야만 응답을 받을 수 있으므로, 처리 과정의 중간 상태(진행률 등)를 전혀 알 수 없습니다.
    *   **낮은 사용자 경험:** 작업 시간이 길어질 경우, 사용자는 응답이 올 때까지 로딩 화면만 보며 기다려야만 합니다.

#### 2. 폴링 (Polling)
*   **정의:** 클라이언트(프론트엔드)가 특정 주기(예: 2초)마다 서버에 진행 상태를 문의하는 요청을 계속해서 보내는 방식입니다. 백엔드는 진행률을 DB에 기록하고, 클라이언트는 이 DB의 값을 주기적으로 확인하여 게이지 바 등을 업데이트합니다.
*   **장점:** 구현이 비교적 간단하고 직관적입니다.
*   **단점:**
    *   **서버/DB 부하:** 실제 데이터 변경이 없어도 계속해서 불필요한 요청을 보내므로 서버와 데이터베이스에 부하를 줍니다.
    *   **지연 발생:** 폴링 주기 때문에 실시간성이 떨어집니다. 최악의 경우, 설정된 주기만큼의 시간 지연이 발생할 수 있습니다.

#### 3. 웹소켓 (WebSockets)
*   **정의:** 클라이언트와 서버 간에 양방향 통신 채널을 한 번만 생성하고, 그 연결을 계속 유지하는 방식입니다. 이를 통해 서버는 데이터가 변경되는 즉시, 능동적으로 클라이언트에게 메시지를 직접 밀어 넣어줄(Push) 수 있습니다.
*   **장점:**
    *   **완벽한 실시간성:** 서버에서 데이터가 변경되는 즉시 클라이언트에 전달되므로 지연 시간이 거의 없습니다.
    *   **효율성:** 불필요한 요청이 없으므로 폴링에 비해 서버 부하가 훨씬 적습니다.
*   **단점:** 폴링에 비해 초기 구현이 복잡할 수 있습니다.

### 서버-클라이언트 데이터 통신 총정리: 직렬화와 역직렬화

#### 1. 핵심 개념: 왜 변환이 필요한가?

네트워크를 통해 서버와 클라이언트가 주고받는 모든 데이터는 근본적으로 **하나의 긴 문자열(텍스트)**입니다.
따라서 서버의 메모리에 있는 데이터 구조(Python `dict`)를 프론트엔드의 JavaScript `Object`로 직접 전달할 수는 없습니다. 이 때문에 두 단계의 변환 과정이 반드시 필요합니다.

*   **직렬화 (Serialization): 서버에서**
    *   메모리 상의 데이터(Python `dict`)를 네트워크로 전송 가능한 형태(JSON 형식의 **문자열**)로 변환하는 과정입니다.
    *   Flask의 `jsonify` 함수가 이 역할을 수행하며, 동시에 `Content-Type: application/json` 헤더를 설정하여 "이 데이터는 JSON 문자열"임을 명시합니다.

*   **역직렬화 (Deserialization): 프론트엔드에서**
    *   서버로부터 받은 JSON 형식의 **문자열**을 JavaScript가 실제로 사용할 수 있는 '살아있는' **객체(Object)**로 재조립하는 과정입니다.

#### 2. 형식(Format)의 중요성: 왜 JSON을 사용하는가?

과거에는 `"박승호|1234|email@naver.com"` 처럼 개발자가 임의의 규칙을 정해 데이터를 직렬화하기도 했습니다. 하지만 이 방식은 데이터를 받는 쪽에서 각 필드의 의미를 미리 알고 있어야만 해석이 가능합니다.

반면, **JSON 형식**은 아래와 같이 데이터의 **의미(key)와 값(value)을 한 쌍으로** 표현하는 표준화된 구조를 가집니다.

`"{ "name": "박승호", "id": 1234, "email": "email@naver.com" }"`

이처럼 데이터 자체가 구조와 의미를 설명해주므로(self-describing), 누가 받더라도 해석하기 용이하여 현대 웹에서 데이터 교환의 표준으로 사용됩니다.

#### 3. 프론트엔드 역직렬화 구현: 두 가지 방식

프론트엔드는 서버로부터 받은 JSON 문자열을 아래와 같은 방식으로 역직렬화하여 JavaScript 객체로 만듭니다.

**A. 최신 방식: `fetch` API (Promise 기반)**

`fetch`는 비동기 작업을 더 깔끔하게 처리하기 위한 최신 기술입니다. `async/await` 문법과 함께 사용하면 코드를 이해하기 쉽습니다.

```javascript
async function fetchData() {
  // 서버에 데이터를 요청하고 응답을 기다림
  const response = await fetch('/get-data-url');
  // 응답 본문의 JSON 문자열을 JavaScript 객체로 역직렬화
  const data = await response.json(); // 이 부분이 역직렬화!

  console.log(data.name); // "박승호"
}
```

**B. 전통 방식: `XMLHttpRequest` (이벤트 기반)**

`XMLHttpRequest`(XHR)는 오랫동안 사용되어 온 전통적인 AJAX 구현 방식입니다.

```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', '/get-data-url');
xhr.send();

xhr.onload = function() {
  if (xhr.status === 200) {
    // 응답으로 온 순수 문자열(xhr.responseText)을 직접 파싱하여 역직렬화
    const data = JSON.parse(xhr.responseText); // 이 부분이 역직렬화!

    console.log(data.name); // "박승호"
  }
};
```
