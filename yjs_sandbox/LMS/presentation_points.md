## 2. 핵심 기능 구현 (Core Features)

### 2.1. 백엔드 (Backend)

*   **Flask 프레임워크 기반 설계:** Python 기반의 Flask를 사용하여 프로젝트의 핵심 로직과 라우팅을 구현, 빠른 개발 속도와 확장성을 확보했습니다.
*   **데이터베이스 연동 (MySQL/MariaDB):** `pymysql` 라이브러리를 통해 사용자 정보, 파일 메타데이터, 분석 결과 등 핵심 데이터를 안정적으로 저장하고 관리합니다.
*   **세션을 이용한 사용자 인증:** Flask의 세션 관리를 통해 사용자의 로그인 상태를 안전하게 유지하고, 인가된 기능에만 접근을 허용합니다.
*   **안전한 파일 업로드 및 관리:**
    *   `secure_filename` 기능을 활용하여 '디렉토리 트래버설'과 같은 보안 위협을 원천적으로 차단합니다.
    *   업로드된 파일은 서버의 `uploads/` 폴더에 안전한 파일명으로 저장되며, 관련 정보는 데이터베이스에 기록되어 체계적으로 관리됩니다.
*   **비동기 처리를 통한 사용자 경험 개선:** AI 분석과 같이 시간이 오래 걸리는 작업은 별도의 백그라운드 스레드에서 비동기적으로 처리하여, 사용자가 분석 중에도 웹 페이지를 지연 없이 사용할 수 있도록 구현했습니다.

### 2.2. 프론트엔드 (Frontend)

*   **템플릿 기반의 페이지 구성:** `templates/` 폴더 내의 각 HTML 파일에 명확한 역할을 부여하여 페이지 구조를 효율적으로 관리합니다.
*   **CSS를 통한 일관된 디자인:** `static/mainstyle.css` 파일을 통해 프로젝트 전체의 디자인 요소를 일관되게 유지하여 통일성 있는 사용자 경험을 제공합니다.
*   **AJAX를 활용한 동적인 UI 구현:** Vanilla JavaScript와 AJAX 기술을 사용하여, 페이지 새로고침 없이 서버와 데이터를 교환하고 AI 분석 상태를 주기적으로 업데이트하여 사용자에게 실시간 피드백을 제공합니다.
*   **데이터 시각화의 중요성 (Chart.js):** (※ 선택적 강조 포인트) 복잡한 AI 분석 결과를 사용자가 쉽게 이해할 수 있도록 Chart.js와 같은 라이브러리를 활용하여 차트와 그래프 형태로 시각화하는 것은 프로젝트의 가치를 크게 높일 수 있습니다.

## 3. 기술적 도전과 해결 (Technical Deep Dive)

### 3.1. 서버-클라이언트 데이터 통신의 핵심: `jsonify`와 직렬화

*   **문제점:** 서버(Python `dict`)와 클라이언트(JavaScript `Object`)는 서로 다른 데이터 구조를 사용하므로 직접 통신할 수 없습니다. 네트워크 통신은 기본적으로 **문자열**로 이루어집니다.
*   **해결: 직렬화 (Serialization)**
    1.  **`jsonify`의 역할:** Flask의 `jsonify`는 Python 딕셔너리를 표준 **JSON 형식의 문자열**로 변환(직렬화)합니다.
    2.  **`Content-Type` 헤더 설정:** 동시에, HTTP 응답 헤더에 `Content-Type: application/json`을 명시하여, 브라우저에게 이 데이터가 JSON임을 정확히 알려줍니다.
*   **결론:** `jsonify`는 단순한 데이터 변환을 넘어, 표준화된 방식으로 서버와 클라이언트가 안정적으로 데이터를 교환할 수 있도록 하는 핵심적인 역할을 수행합니다.

### 3.2. 실시간 피드백 구현을 위한 비동기 방식 비교 및 선택

*   **고려사항:** AI 분석과 같이 오래 걸리는 작업의 진행률을 사용자에게 실시간으로 보여주어야 했습니다.
*   **선택지 비교:**
    *   **폴링 (Polling):** 클라이언트가 주기적으로 서버에 "끝났어?"라고 물어보는 방식. 구현은 간단하지만, 불필요한 요청이 많아 서버에 부하를 줍니다.
    *   **웹소켓 (WebSockets):** 서버와 클라이언트 간의 양방향 통로를 열어두고, 서버가 필요할 때마다 클라이언트에게 "진행률 50%야!" 라고 능동적으로 알려주는 방식. 실시간성이 뛰어나지만, 폴링에 비해 초기 구현이 복잡합니다.
*   **우리의 선택: 폴링 (Polling)**
    *   프로젝트의 현재 규모와 요구사항을 고려했을 때, **폴링 방식**은 실시간 피드백 기능을 **가장 효율적이고 직관적으로 구현**할 수 있는 합리적인 선택이었습니다. `setInterval` 함수를 사용해 간단하게 구현하고, 서버 부하 문제는 향후 트래픽 증가 시 웹소켓으로 전환하는 방식으로 대응할 수 있습니다.

### 3.3. 기술 선택의 이유: 왜 Vanilla JavaScript와 AJAX인가?

*   **배경:** 초기에는 `Fetch API` 등 최신 기술을 고려했으나, 논의 끝에 순수 JavaScript와 전통적인 AJAX(`XMLHttpRequest`)를 핵심 기술로 채택했습니다.
*   **결정의 이유:**
    1.  **팀워크 및 협업 극대화:** 모든 팀원이 기술 스택에 동등하게 기여하고 함께 성장할 수 있는 환경을 조성했습니다.
    2.  **기본기 강화:** 특정 라이브러리에 의존하기보다 JavaScript의 근본적인 동작 원리와 비동기 통신의 핵심을 더 깊이 이해하는 계기가 되었습니다.
    3.  **유연성 및 경량화:** 별도의 라이브러리 없이 브라우저 호환성을 확보하고 가벼운 웹페이지를 구현했습니다.
*   **결론:** 이 결정은 우리 팀의 유연한 의사소통과 기술의 본질에 집중하는 개발 철학을 보여주는 좋은 사례입니다.
