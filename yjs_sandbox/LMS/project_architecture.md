# 프로젝트 아키텍처 및 흐름도 (텍스트 기반)

## 1. 개요

본 문서는 Flask 기반 웹 애플리케이션의 전체적인 아키텍처, 데이터 흐름, 그리고 주요 기능별 동작 원리를 설명합니다. 이 문서는 프로젝트의 구조와 흐름을 쉽게 파악하여 발표 자료 등으로 활용할 수 있도록 텍스트 중심으로 작성되었습니다.

## 2. 기술 스택

*   **백엔드 프레임워크:** **Flask**
    *   Python 기반의 마이크로 웹 프레임워크로, `app.py`가 핵심 라우팅 및 비즈니스 로직을 처리합니다.
*   **프론트엔드:**
    *   **HTML/CSS:** `templates/` 폴더의 HTML 파일들이 페이지 구조를 정의하고, `static/mainstyle.css`가 전체적인 디자인과 레이아웃을 담당합니다.
    *   **JavaScript:** 서버와의 비동기 통신(AJAX) 및 동적인 UI 제어를 위해 Vanilla JavaScript가 일부 사용됩니다.
*   **데이터베이스:** **MySQL**
    *   `pymysql` 라이브러리를 통해 Python 코드와 상호작용하며, 사용자 정보, 미디어 파일 메타데이터, 분석 결과 등을 영구적으로 저장합니다.

## 3. 전체 시스템 동작 흐름

이 시스템은 사용자의 요청부터 서버의 처리, 그리고 최종 응답까지 다음과 같은 단계로 동작합니다.

1.  **사용자 요청 (Client-Side):**
    *   사용자가 웹 브라우저를 통해 특정 URL(예: `/login`, `/analyze`)에 접속하면, 해당 주소로 HTTP 요청이 발생합니다.

2.  **웹 서버 수신 및 라우팅 (Server-Side - Flask):**
    *   Flask 애플리케이션(`app.py`)이 이 요청을 수신합니다.
    *   요청된 URL 경로를 보고, 미리 정의된 라우트(`@app.route`) 규칙에 따라 어떤 Python 함수를 실행할지 결정합니다.

3.  **비즈니스 로직 처리:**
    *   **데이터베이스 연동:** 실행된 함수는 필요시 `common/Session.py`의 `get_connection()`을 호출하여 데이터베이스에 연결합니다. 이후 SQL 쿼리를 통해 데이터를 조회(SELECT), 생성(INSERT), 수정(UPDATE), 삭제(DELETE)하는 작업을 수행합니다.
    *   **세션 및 인증:** 로그인이 필요한 기능의 경우, Flask의 `session` 객체에 `user_id` 정보가 저장되어 있는지 확인하여 사용자의 로그인 상태를 검증합니다.
    *   **파일 처리:** 사용자가 파일을 업로드하면, 서버는 파일을 `uploads/` 폴더에 안전한 이름으로 저장하고, 관련 정보를 데이터베이스에 기록합니다.

4.  **비동기 백그라운드 작업 (AI 분석):**
    *   파일 업로드와 같이 시간이 오래 걸리는 AI 분석 작업은 별도의 **백그라운드 스레드(Thread)**에서 처리됩니다.
    *   이를 통해 메인 서버는 즉시 사용자에게 '처리 중'이라는 응답을 보낼 수 있으며, 사용자는 분석이 완료될 때까지 기다릴 필요가 없습니다.
    *   백그라운드 스레드는 작업이 완료되면 데이터베이스의 상태 값을 `PENDING`에서 `SUCCESS`로 업데이트합니다.

5.  **템플릿 렌더링 및 응답:**
    *   비즈니스 로직 처리가 완료되면, 서버는 `render_template()` 함수를 사용합니다.
    *   이 함수는 지정된 HTML 파일(`templates/` 폴더 내)에 동적으로 처리된 데이터(예: DB에서 가져온 사용자 정보, 파일 목록)를 삽입하여 최종 HTML 코드를 생성합니다.
    *   생성된 HTML은 HTTP 응답(Response)의 형태로 다시 사용자 브라우저로 전송됩니다.

6.  **사용자 화면 표시:**
    *   브라우저는 서버로부터 받은 HTML 응답을 해석하여 사용자에게 웹 페이지 화면을 보여줍니다. 이 과정에서 필요한 CSS나 이미지 같은 정적 파일들도 추가로 서버에 요청하여 받아옵니다.

## 4. 기능별 동작 원리 상세

### 4.1. 사용자 인증 흐름

*   **회원가입:** 사용자가 입력한 정보를 받아 아이디 중복을 검사한 후, `members` 테이블에 새로운 사용자를 추가합니다.
*   **로그인:** 사용자가 제출한 아이디/비밀번호를 `members` 테이블과 대조하여 일치하는 레코드가 있는지 확인합니다. 인증에 성공하면, 해당 사용자의 고유 ID, 이름, 역할 등을 암호화된 `session` 쿠키에 저장하여 클라이언트에 발급합니다. 이후의 모든 요청에는 이 세션 정보가 포함되어 서버가 누구인지 식별할 수 있습니다.
*   **로그아웃:** `session` 객체의 모든 데이터를 삭제하여 인증 상태를 해제합니다.

### 4.2. 미디어 파일 분석 기능 흐름

이 기능은 전통적인 게시판의 CRUD(Create, Read, Update, Delete) 패턴을 따릅니다.

*   **생성 (Create):** 파일 업로드 시, 서버는 파일을 `uploads/` 폴더에 저장하고(`mediafile_uploads` 함수), 해당 파일의 메타데이터(소유자, 파일명, 저장경로 등)를 `media_files` 테이블에, 분석 상태를 `analysis_results` 테이블에 각각 기록합니다.
*   **조회 (Read):**
    *   **목록 조회 (`/analyze/list`):** 현재 로그인한 사용자의 모든 미디어 파일 목록을 DB에서 조회하여 화면에 표 형태로 보여줍니다.
    *   **상세 조회 (`/analyze/analysis/<id>`):** 특정 파일의 상세 정보와 함께, `analysis_results` 테이블에서 AI 분석 결과를 가져와 포맷팅하여 보여줍니다.
*   **삭제 (Delete):** 사용자가 파일을 삭제하면, 서버는 먼저 `uploads/` 폴더에서 실제 파일을 삭제하고, 그 다음 `media_files`와 `analysis_results` 테이블에서 관련 데이터 레코드를 모두 삭제하여 작업을 완료합니다.

### 4.3. 비동기 AI 분석 처리 흐름

*   **요청과 분리:** 파일이 업로드되면, 분석 작업은 즉시 실행되지 않고 백그라운드 스레드로 위임됩니다. 이로써 사용자는 웹 페이지에서 지연 없이 다른 작업을 계속할 수 있습니다.
*   **상태 관리:** 분석 작업의 진행 상태는 `analysis_results` 테이블의 `status` 컬럼(`PENDING`, `SUCCESS`, `FAIL`)을 통해 관리됩니다. 프론트엔드에서는 이 상태 값을 주기적으로 API로 확인하여 사용자에게 실시간 진행 상황을 보여줄 수 있습니다.
*   **결과 저장:** 분석 스레드는 작업이 끝나면, 결과 데이터를 JSON 형태로 `result_json` 컬럼에 저장하고 상태를 `SUCCESS`로 업데이트합니다. 사용자는 이후 상세 조회 페이지에서 이 최종 결과를 확인할 수 있습니다.
