## 4\. 시스템 아키텍처 (설계도) 소개

"본격적인 기술 및 모델 설명에 앞서, 저희가 구축한 전체 시스템 아키텍처를 보여드리겠습니다."
"(아키텍처 슬라이드를 띄우며)"

"저희 아키텍처에서 가장 신경 쓴 부분은 일반 웹 서버(Flask)와 무거운 AI 분석 모델의 **퍼포먼스 분리**였습니다."

"사용자가 화면에서 영상과 요청을 보내면(Input), 메인 서버는 요청과 파일을 안전한 저장소로 분리하여 관리합니다."

"AI 모델이 뒷단에서 무거운 영상을 분석하는 동안, 프론트엔드는 **XMLHttpRequest 기반의 클라이언트 비동기 대기 방식**을 채택하였습니다. 이를 통해 응답이 완료될 때까지 상태를 안정적으로 유지하고, 분석이 완료되면 최종 결과물을 로컬 캐시를 거쳐 사용자 화면에 깔끔하게 보여주는(View Render) 견고한 아키텍처를 설계하였습니다."

\---

## 5\. DB 설계 소개

"저희는 관계형 데이터베이스인 MySQL을 사용하였으며, 총 네 개의 테이블로 서비스 데이터를 관리합니다."

"스키마 설계 시 **3NF(제3정규형)** 수준을 기준으로 삼았습니다. 회원 정보, 미디어 파일, 분석 결과, 게시글을 각각의 테이블로 분리하여 이행적 종속이 발생하지 않도록 하였고, 각 테이블의 모든 속성은 원자값으로만 구성하여 **1NF** 요건도 준수하였습니다."

"또한 **외래키와 CASCADE 삭제 규칙**을 적극 활용하여, 참조 무결성을 애플리케이션 코드가 아닌 데이터베이스 레벨에서 직접 보장하도록 하였습니다. 한 가지 예외적인 설계 결정은 AI 탐지 결과입니다. 탐지된 객체의 좌표, 태그, 신뢰도 등은 구조가 가변적이어서 정규화보다 유연성이 우선되는 상황이라 판단하여, **JSON 타입 컬럼**에 비정규 형태로 저장하는 방식을 의도적으로 선택하였습니다."

\---

## 6\. 프론트엔드 소개

"아키텍처와 연결하여, 저희 프론트엔드가 어떤 방식으로 구현되어 있는지 세 가지 핵심 개념을 중심으로 설명드리겠습니다."

"첫 번째는 **AJAX Communication**, 즉 비동기 통신 최적화입니다. 저희는 XMLHttpRequest를 활용하여 화면 새로고침 없이 실시간으로 분석 요청을 보내고 응답을 받는 체계를 구축하였습니다. 덕분에 사용자는 분석이 완료될 때까지 페이지 이동 없이 자연스럽게 결과를 기다릴 수 있습니다."

"두 번째는 **Dynamic Rendering**, 조건부 뷰 제공입니다. Session 상태를 실시간으로 감지하여, 로그인 여부와 사용자 권한에 따라 맞춤형 UI를 렌더링합니다. 예를 들어 로그인한 사용자에게는 마이페이지와 분석 이력이 보이고, 비로그인 사용자에게는 로그인 유도 화면이 나타나도록 설계하였습니다."

"세 번째는 **Data Validation**, 안전한 데이터 이동입니다. 파일 형식, 용량, 입력값 등을 서버로 전송하기 전 프론트엔드 단에서 먼저 검사합니다. 이를 통해 불필요한 서버 부하를 줄이고, 무결성이 보장된 데이터만 백엔드로 전달되도록 하였습니다."

"이 세 가지를 통해 Flask 기반 웹 연동과 비동기 UX를 구현하였고, 사용자 경험 중심의 가볍고 빠른 인터페이스를 지향하였습니다."

## 7\. 단계별 기술 상세: 모델 훈련 및 성능 지표

"이전 저희 프로젝트에서 핵 역할을 담당하는 AI 모델에 대해 설명드리겠습니다. 무거운 환경을 고려해 속도와 정확도 밸런스가 뛰어난 YOLOv11 미디엄(Medium) 모델을 backbone으로 삼았습니다."

"(슬라이드의 파라미터/훈련 코드를 가리키며) 화면에 보이시는 것처럼, 저희는 입력 해상도를 640으로 설정하고, 배치 사이즈 16으로 총 50 Epoch을 돌리는 훈련을 진행하였습니다. 추가로 디바이스는 cuda를 사용할 수 있을 시 cuda로 설정해 gpu를 사용하도록 조정하였습니다."

## 8\. 단계별 기술 상세: 서버 최적화 및 경량화 (한계극복)

"하지만 아무리 뛰어난 모델이라도, 이 무거운 원본 파이토치 웹 모델(.pt)을 그대로 올려 이용한다면 서버 환경에서는 필연적으로 과부하가 발생하며, 그 성능을 제대로 끌어올리지 못해 결과적으로 원본 모델에 미치지 못하는 성능을 내는 현상을 겪게 되었습니다."

"그래서 서버 성능에 맞춘 백엔드 최적화 전략, 즉 경량화 기법을 자체적으로 적용하였습니다."

"(코드 블록 가리키며) 서버를 가동할 때 무거운 파이토치 파일을 그대로 쓰지 않고, 곧바로 가벼운 포맷인 ONNX 환경으로 자체 변환하도록 로직을 짰습니다. 특히 `half=True` 파라미터를 사용해 데이터를 표현하는 비트 수를 반으로 자르는 **'FP16 경량화(Quantization)'** 기법을 적용해 서버용으로 가볍게 다이어트시켰습니다."

"이를 통해 무거운 연산 부담을 제거하고, 탐지 성능을 어느 정도 유지하면서 일반적인 웹 서버에서도 부드럽게 구동되는 맞춤형 백엔드를 완성하였습니다."

## 9\. 실제 구동 시연

"이러한 원칙을 바탕으로, 최적화된 백엔드 서버와 모델을 올려 실제 테스트를 진행해 보겠습니다."

(실제 시연 순서)
1. 사이트 소개
2. 회원가입 - 비동기 실시간 db 현황 가져오기 - 이메일 형식 틀릴 시 서브밋 되돌리기 - 비밀번호 확인 틀릴 시 서브밋 되돌리기
3. 로그인 - 아이디 찾기 기능
4. 마이 페이지 - 정보 수정 - 프로필과 한마디
5. 분석 게시판- 자신의 분석 게시판만 확인 가능(추후 비밀 분석 토글을 만들어서 퍼블릭 or 프라이빗 2분 가능성 검토)
6. 이미지 실제 분석 시도 - 비동기 로직으로 화면을 넘어가지 않게 하고 분석가능 - 분석 완료시 분석 결과창으로 자동 이동
7. 영상 분석 - 영상 분석은 오래 걸리므로 미리 준비된 psh 계정 영상을 활용
8. 관리자 페이지 - 전체 회원/전체 분석 게시글/전체 문의 게시글 확인 가능 - 문의 게시글은 직접 수정해서 답변 가능

## 부록: 예상 Q\&A 대비

### ex. 만약 data.yaml에 대한 질문이 들어온다면?

"네, 질문에 답변드리겠습니다. data.yaml 파일은 쉽게 말해 AI 모델에게 데이터가 어디에 있는지, 무엇을 학습해야 하는지 알려주는 '안내 지도(설정 파일)'입니다."

**(조금 더 기술적으로 말붙인다면)**
"저희가 구급차와 일반 차량 이미지를 매겨 폴더에 정리해 두었는데요. YOLO 모델은 이 폴더들이 컴퓨터의 어느 경로에 있는지 스스로 알지 못합니다. 그래서 이 data.yaml 파일 안에 세 가지를 지정해 줍니다.

1. 훈련용 이미지와 검증용 이미지가 있는 '폴더 경로'
2. 우리가 맞춰야 할 정답의 개수 (클래스 수: nc)
3. 각 정답의 텍스트 이름 (names: \['ambulance', 'car'])
모델은 학습을 시작할 때 가장 먼저 이 yaml 파일을 읽고 자기가 공부할 데이터를 찾아오게 됩니다."
